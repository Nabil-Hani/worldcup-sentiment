# ⚽ Real-Time World Cup Sentiment Tracker

A production-grade NLP dashboard that streams simulated match-day tweets through a pre-trained RoBERTa model and visualises live sentiment in a Plotly Dash app.

---

## Architecture

```
worldcup-sentiment/
├── app/
│   ├── app.py          ← Dash layout + Interval callbacks (entry-point)
│   ├── simulator.py    ← Generator that "streams" tweets from CSV
│   ├── sentiment.py    ← HuggingFace inference wrapper
│   └── state.py        ← Thread-safe ring-buffer shared between threads
├── data/
│   └── raw/
│       └── tweets.csv  ← Generated by generate_mock_data.py
├── models/             ← (Optional) place fine-tuned model checkpoints here
├── notebooks/          ← Exploratory analysis / EDA notebooks
├── generate_mock_data.py
├── requirements.txt
└── README.md
```

## Quickstart

```bash
# 1. Clone and install
git clone https://github.com/YOUR_USERNAME/worldcup-sentiment.git
cd worldcup-sentiment
pip install -r requirements.txt

# 2. Generate mock tweet data (one-time)
python generate_mock_data.py

# 3. Launch the dashboard
python app/app.py
# → Open http://localhost:8050
```

## Component Deep-Dive

### `simulator.py` — The Stream Generator

Uses a Python **generator function** (`yield`) to emit tweets one at a time with a configurable `time.sleep()` interval. This pattern is identical in structure to consuming a real streaming API — swap `tweet_stream()` for a `tweepy.StreamingClient` callback and the rest of the pipeline is unchanged.

```python
for tweet in tweet_stream(interval=1.5):
    process(tweet)
```

### `sentiment.py` — NLP Inference

Wraps `cardiffnlp/twitter-roberta-base-sentiment-latest`, a RoBERTa model trained on **124 million tweets** and fine-tuned for 3-class sentiment. The pipeline is lazy-loaded via `@lru_cache` so the ~500 MB model is only downloaded and parsed **once** per process lifetime.

| Output Label | Numeric Score | Meaning |
|---|---|---|
| Positive | +1 | Joy, celebration, praise |
| Neutral | 0 | Factual updates, half-time scores |
| Negative | −1 | Frustration, criticism, disappointment |

### `state.py` — Thread-Safe Buffer

The simulator runs in a background **daemon thread**; Dash callbacks run in the main thread. `SentimentStore` uses a `threading.RLock` and a `collections.deque(maxlen=200)` to pass data between them without race conditions. The `maxlen` cap bounds memory usage during long demo sessions.

### `app.py` — Dash Dashboard

The `dcc.Interval` component fires every **3 seconds**, triggering a single `@callback` that:
1. Reads a snapshot from `SentimentStore`
2. Rebuilds all four visual components (KPI cards, timeline, donut, tweet feed)

All layout is constructed in pure Python — no external CSS files needed.

## Why These Choices?

| Decision | Reasoning |
|---|---|
| `twitter-roberta-base-sentiment-latest` | Trained on Twitter text which has unusual casing, slang, and emoji — far better than general-purpose BERT on this domain |
| `dcc.Interval` (not WebSockets) | Simpler architecture; polling every 3s is indistinguishable from "live" for human viewers |
| `deque(maxlen=200)` | Bounded memory — the dashboard stays fast during a 90-minute match simulation |
| Generator with `time.sleep()` | Decouples stream speed from inference speed; easy to tune `interval` without touching the NLP layer |
| Daemon thread | Automatically killed when the main process exits — no cleanup code needed |

## Extending the Project

- **Real API**: Replace `tweet_stream()` with a `tweepy.StreamingClient` consumer. The `analyze()` and `store.append()` calls are unchanged.
- **Fine-tuning**: Label a subset of `tweets.csv` using `true_label` and fine-tune the RoBERTa head with `transformers.Trainer` for football-specific vocabulary.
- **Persistence**: Swap `SentimentStore` for a SQLite write + `pd.read_sql` in the callback to survive app restarts.
- **Deployment**: `gunicorn app.app:server --workers=1 --threads=4` on any Linux VM; add a Dockerfile for container deployment.
